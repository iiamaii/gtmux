§B. 실제 결함 (P0)

P0실제 결함B-1. reconnectGate.cancel() 가 tentative attach 의 BE lock 을 release 안 함

📍 위치

codebase/frontend/src/lib/stores/reconnectGate.svelte.ts:111-116

cancel(): void {
  this.#controller?.abort();
  this.#controller = null;
  this.markIdle();
  sessionStorageHint.clear();
  // ❌ DELETE /api/sessions/<attemptName>/attach 호출 없음
}

🔁 재현 시나리오

  1. 브라우저: sessionStorage.gtmux-last-active-session = "alpha"
  2. 페이지 reload → AppPage.onMount 가 hint 발견 → reconnectGate.start("alpha") → POST /api/sessions/alpha/attach
  3. BE attach_handler 가 flock 잡고 lease body 쓰는 도중 또는 200 응답 직전
  4. 사용자가 ReconnectModal 의 [Switch session…] 클릭
  5. cancel() 호출 → AbortController.abort() → in-flight fetch 취소
  6. BE 는 이미 200 응답 직전이거나 응답을 보냈는데 FE 가 못 받은 상태 — 서버 측 flock + reverse-map 잡혀있음
  7. FE 는 SessionListModal 로 이동 (state-machines.md §3.3 의 'choice' routing)

📊 영향

  • 자가 회복 케이스 (다행): 같은 owner_key 로 "alpha" 재시도 → idempotent 200 (sessions.rs:461-466)
  • 회복 안 되는 케이스 (문제): 다른 webpage_id 의 새 탭으로 "alpha" 진입 → 409 + “in use” badge → 30s heartbeat timeout 까지 lock 잔존. 사용자 perception: “내가 cancel 했는데도 session 이 잡혀있다”
  • 회복 안 되는 케이스 (보안): cookie 만료 직전 cancel → 30s 보다 더 오래 lock 잔존 가능 (작은 race)

🔄 대비 사례

WorkspaceSwitcher.svelte:215-250 의 cancelAttachConfirm 은 명시적으로 detachSession(pending) 호출 (state-machines.md §3.2.1 의 step 1, ADR-0019 D5.5.1). reconnectGate 만 이 패턴에서 누락.

📜 ADR 정합성

ADR-0019 D5.4 본문 + state-machines.md §3.3 의 “ReconnectModal [Switch session…]” 행은 명시 cancel = lock release 의도가 직접 명시되어 있지 않음. → 이 비-amend 가 결함의 근원.

✅ 권장 fix

  1. (즉시 land, FE-only) reconnectGate.cancel() 에 state === 'attaching' && attemptName !== null 분기에 best-effort detachSession(attemptName) 호출. 실패는 silent — 30s heartbeat fallback. 비동기 fire-and-forget OK
  2. (짝 commit, ADR amend) ADR-0019 D5.4 amend ② — “사용자 명시 cancel = tentative lock release. detach 실패는 30s heartbeat 가 fallback” 명시

P0실제 결함B-2. “terminal history 미표시” — by-design 함정 + UX 미고지

🎯 근본 원인

PaneId 는 process-bound. 서버 재시작 시 모든 PaneId 가 fresh, 옛 process 의 ring buffer 가 OS-level 로 사라짐. terminal_map 의 UUID↔PaneId binding 도 in-memory 라 사라짐. AttachConfirm 흐름이 새 UUID→PaneId 를 새 process 로 binding 하지만 새 process 의 ring buffer 는 0 bytes.

🔍 현 동작 (verified)

ws-server/lib.rs:613-686 의 catch-up replay 흐름:

1. TerminalSpawned 0x88 burst — 모든 alive (PaneId, UUID) pair. FILTER 없음
2. session_pane_set 가 cookie-attached 면 cold-load (lib.rs:573-583)
3. for each id in backend.pane_ids():
     if !session_pane_set.contains(id.0) { continue; }    ← amend ③ filter
     send NOTIFY pane-spawned
     send PANE_OUT (ring buffer replay, backend.subscribe_output(id).0)
  • 서버 재기동 직후: backend.pane_ids() 가 empty → catch-up replay 가 단 1 byte 도 안 보냄
  • AttachConfirm 후 respawn 시점: live broadcast terminal_spawned_events 가 새 binding emit → FE terminalPool 가 binding 추가 → panel xterm mount → 새 process 의 fresh shell

🧠 왜 “꼬여보이는가” (사용자 mental model)

  • 사용자: “session 은 file 영속이니 history 도 영속이어야지” — ADR-0019 D5 의 session file 영속 과 terminal process 영속 의 분리를 인지 못 함
  • ADR-0021 D10 (lifecycle) 은 process die = history die 를 명시. 그러나 UX layer 에서 이 사실이 노출 안 됨
  • AttachConfirmModal 의 현 copy: “Will spawn N new terminal(s). Continue?” — 이전 출력은 복원 불가 라는 경고 없음

⚠️ 잠재적 부가 원인 (verification 필요)

ADR-0021 D8 의 [Attach existing terminal] (Terminal list → 현 session 으로 mount):

  • 이 경우 PaneId 는 alive — ring buffer 가 BE 메모리에 있음
  • FE PUT /layout 로 새 panel item 추가
  • BE 가 layout_events broadcast → WS의 session_pane_set hot-update 가 PaneId 추가
  • 그러나 backend.subscribe_output() 의 replay tuple 은 catch-up 시점에만 보내짐 — hot-update path 에는 ring buffer 의 그 시점 replay 가 포함 안 됨
  • 사용자 perception: “이 terminal 에 분명 history 가 있는데 attach 후 빈 화면이다”
verify 필요: 이게 verify 되면 두 번째 P0 — 코드 fix 가 필요한 진짜 결함. 본 감사에서 코드 경로만 식별, 재현 시연 안 함. 다음 manual E2E 에서 확인.

✅ 권장 fix — 3 옵션

  1. (a) UX-only (저비용): AttachConfirmModal 의 copy 에 명시 경고 — “기존 terminal 들은 새 process 로 spawn 됩니다 — 이전 output 은 복원되지 않습니다.” + ADR-0018 D6, ADR-0021 D10 본문 amend
  2. (b) Attach-time ring replay: ADR-0021 D8 amend — [Attach existing terminal] 흐름에 그 시점의 ring buffer 1회 replay. 새 endpoint POST /api/terminals/<uuid>/replay 또는 PUT 응답 envelope 동봉
  3. (c) Auto-replay on layout PUT: put_layout_handler 의 added UUIDs 의 ring buffer 를 그 session 의 WS 로 emit. 첫 추가만 dedup 필요

권장: (a) 즉시 land + (b) ADR amend draft → 사용자 결정 후 구현.

A. 본 감사의 조사 방법론

  1. Phase 1 — ADR/SSoT 정독: state-machines.md → ADR-0019 → ADR-0020 → ADR-0021 → ADR-0025. 합성 SoT 우선, 정본 ADR cross-check
  2. Phase 2 — BE/FE 병렬 정찰: Explore agent 2 개 spawn, ADR ↔ 코드 정합 1차 보고

B. 본 감사가 완전 검증 안 한 영역

  • D-4 cancel chain toast: 0069 가 명시한 8s warning toast 의 실제 출력 trace 안 함
  • B-2.4 rebind history replay: 코드 경로만 식별, 재현 시연 안 함. 다음 manual E2E 에서 확인
  • WS subprotocol 의 webpage_id 누락 영향: HTTP 만 header, WS 는 query — 의도된 분리 같으나 invariant ADR 명시 없음
  • multi-cookie scenario: 같은 호스트의 multi-user 가 아닌 multi-tab 만 검토. ADR scope 가 single-user 라 자연 OK
  • /api/shutdown 의 정상 종료 vs SIGKILL 차이: 두 path 의 boot 회복 검증 안 함
